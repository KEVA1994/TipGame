# Implementeringsplan: Multi-turnerings-platform

**Mål:** TipGame ombygges fra én hardcodet turnering (VM 2026) til en platform,
hvor enhver bruger kan oprette en tipskonkurrence, blive admin for den, vælge
turnering (VM, EM, Premier League, …), invitere deltagere via link og styre
mailudsendelser — uden at der skal røres ved kode, secrets eller pg_cron.

**Status i dag (udgangspunkt):**

- Skema: `Users` (Id, Name, AuthId), `Matches` (uden turnerings-kobling),
  `Predictions`, `SentReminders`. Alt er implicit "den ene turnering".
- RLS er reelt slået fra: `Users` og `Predictions` har `ALL`-policies med
  `qual: true` (alle kan skrive alt), `Matches` er public read.
- Edge-funktionen `sync-matches` har `WC`-endpointet hardcodet og styres af
  secrets + manuel pg_cron-opsætning (pt. afmeldt, se REAKTIVERING.md).
- Mails (påmindelser + afslutning) er Node-scripts i GitHub Actions, hardcodet
  til "VM Tips Kuponen 2026", og kan kun udløses af repo-ejeren.

## Etape 1 — Datamodel og migrering af VM-data

### Nye tabeller

```sql
create table "Competitions" (
  "Id" int generated by default as identity primary key,
  "Name" text not null,                       -- fx "Arps EM-kupon 2028"
  "CompetitionCode" text,                     -- football-data.org-kode: WC, EC, PL, ... (null ved manuel)
  "Source" text not null default 'football-data',  -- 'football-data' | 'manual'
  "DateFrom" date,                            -- API-hentevindue (null = hele turneringen)
  "DateTo" date,
  "Status" text not null default 'active',    -- 'active' | 'finished' | 'archived'
  "InviteToken" uuid not null default gen_random_uuid(),
  "RemindersEnabled" boolean not null default true,
  "CreatedAt" timestamptz not null default now(),
  constraint "UQ_Competitions_InviteToken" unique ("InviteToken")
);

create table "CompetitionMembers" (
  "Id" int generated by default as identity primary key,
  "CompetitionId" int not null references "Competitions"("Id") on delete cascade,
  "UserId" int not null references "Users"("Id") on delete cascade,
  "Role" text not null default 'player',      -- 'admin' | 'player'
  "JoinedAt" timestamptz not null default now(),
  constraint "UQ_CompetitionMembers" unique ("CompetitionId", "UserId")
);
```

### Ændringer i eksisterende tabeller

- `Matches` får `CompetitionId int not null references "Competitions"` og den
  globale unique på `ExternalId` erstattes af unique på
  `("CompetitionId", "ExternalId")`. Hver konkurrence ejer sine egne
  kamp-rækker — også hvis to konkurrencer følger samme turnering. Det koster
  duplikerede rækker, men gør isolation, RLS og pointberegning triviel.
- `Predictions` og `SentReminders` er allerede match-scoped og behøver ingen
  ændringer (konkurrencen følger med via kampen).
- `Users.Name` er i dag globalt — den bevares som default-navn; visningsnavn
  pr. konkurrence er en mulig senere udvidelse (ikke i denne omgang).

### Migrering af eksisterende data (i samme migration)

1. Indsæt konkurrencen `VM Tips Kuponen 2026` med `Status='finished'`,
   `CompetitionCode='WC'`.
2. Sæt `CompetitionId` på alle 104 eksisterende kampe.
3. Indsæt alle 28 brugere i `CompetitionMembers` (Kasper Arp som `admin`,
   resten `player`).
4. Først derefter sættes `Matches.CompetitionId` til `not null`.

Dermed er den gamle turnering et fuldgyldigt (arkiveret) eksempel på det nye
system — historikken kan stadig ses i appen.

### settle_points

RPC'en beregner point pr. kamp og er allerede turnerings-agnostisk (den følger
`MatchId`). Den skal kun verificeres, ikke ændres.

## Etape 2 — Join-flow og RPC'er (SECURITY DEFINER)

Invitationslinket må ikke kræve, at anonyme kan læse `Competitions`-tabellen
(tokenet skal ikke kunne enumereres). Derfor sker oprettelse/join via RPC'er:

```text
create_competition(name, code, source, date_from, date_to) -> competition_id
  – opretter konkurrencen OG membership med Role='admin' for kalderen (atomisk).

get_competition_by_token(token) -> (id, name, status)
  – slår kun op på eksakt token; bruges af join-siden til at vise "Du er
    inviteret til X" før login/signup.

join_competition(token) -> competition_id
  – kræver login; finder/opretter Users-rækken ud fra auth.uid() og indsætter
    membership som 'player'. Idempotent (allerede medlem = OK).

regenerate_invite_token(competition_id) -> nyt token   (kun admin)
```

Alle fire er `security definer` med eksplicitte tjek på `auth.uid()`, så de
virker uafhængigt af RLS-policies.

**Kobling bruger ↔ auth:** i dag oprettes `Users`-rækken løst koblet via
`AuthId`. `join_competition` overtager ansvaret: findes der ingen `Users`-række
for `auth.uid()`, oprettes den med `display_name` fra auth-metadata.

## Etape 3 — RLS-stramning (deployes SIDST, se rækkefølge)

Nuværende policies er åbne (`qual: true` for ALL på Users/Predictions) — det
var OK i én privat venneliga, men ikke med flere konkurrencer. Nye policies:

Hjælpefunktion: `current_user_id()` → `Users.Id` for `auth.uid()`
(`stable`, `security definer`).
Hjælpefunktion: `is_member(competition_id)` / `is_admin(competition_id)`.

| Tabel | SELECT | INSERT/UPDATE/DELETE |
|-------|--------|----------------------|
| `Competitions` | medlemmer | kun admin (UPDATE af Name/Status/RemindersEnabled); INSERT kun via RPC |
| `CompetitionMembers` | medlemmer af samme konkurrence | admin må DELETE (smide ud); INSERT kun via RPC |
| `Matches` | medlemmer af kampens konkurrence | ingen (kun service role / edge function) |
| `Predictions` | medlemmer af kampens konkurrence | egen række (`UserId = current_user_id()`), og kun mens kampen ikke er startet: `exists (select 1 from "Matches" m where m."Id" = "MatchId" and m."KickoffTime" > now() and m."Status" in ('SCHEDULED','TIMED'))` |
| `Users` | brugere man deler mindst én konkurrence med + sig selv | egen række (Name) |
| `SentReminders` | ingen (kun service role) | ingen |

Bonus: kickoff-reglen på `Predictions` håndhæves i dag kun i klienten — den
bliver nu en databasegaranti.

## Etape 4 — Edge-funktionen: sync pr. aktiv konkurrence

`sync-matches/index.ts` ombygges:

```text
1. select * from Competitions where Status='active' and Source='football-data'
2. Ingen rækker -> return { skipped: true }  (gratis no-op — cron kan køre altid)
3. For hver konkurrence:
   a. GET api.football-data.org/v4/competitions/{code}/matches[?dateFrom&dateTo]
   b. Upsert kampe med CompetitionId (samme feltlogik som i dag:
      ScoreLocked-respekt, aldrig un-finish, regularTime frem for fullTime)
   c. Slut-detektion pr. konkurrence (som i dag: finale spillet + alt færdigt
      => opdater Competitions.Status='finished')
4. Kald settle_points én gang til sidst
```

- **Rate limit:** football-data.org gratis-tier tillader 10 kald/min. Med cron
  hvert minut og N aktive konkurrencer bruges N kald/min. Guard: hvis
  `antal aktive konkurrencer > 8`, syncses i round-robin (kolonne
  `LastSyncedAt`, tag de ældste først). I praksis rammes det næppe.
- **pg_cron sættes op én gang og røres aldrig igen** — dét fjerner hele
  "genaktiverings"-ritualet fra REAKTIVERING.md.
- Secrets `FOOTBALL_API_COMPETITION`, `FOOTBALL_API_DATE_FROM/TO` udgår;
  kun `FOOTBALL_API_TOKEN` beholdes.

**Manuel kilde (`Source='manual'`):** dækker Superligaen og alt andet uden for
football-data.org's 12 gratis turneringer. Admin opretter kampe og indtaster
resultater i admin-UI'et (kræver en `insert/update`-RPC til kampe, kun admin,
med `settle_points`-kald efter resultatindtastning). Sync-funktionen springer
dem bare over.

## Etape 5 — Blazor-UI

### Ny tilstand: valgt konkurrence

- `CompetitionState`-service (som `PlayerState`): liste over "mine
  konkurrencer", valgt konkurrence gemt i localStorage, `OnChange`-event.
- Konkurrence-vælger i toppen af layoutet (dropdown, når man er med i >1).
- Er man ikke medlem af nogen → landing-side med "Opret konkurrence" /
  "Har du fået et link? Bed om et nyt fra din admin."

### Eksisterende sider (får `CompetitionId`-filter fra `CompetitionState`)

`Home`, `Matches`, `MatchDetail`, `Leaderboard`, `PlayerDetail`, `Stats` —
alle queries i `MatchService`, `LeaderboardService`, `PredictionService`,
`StatsService` filtreres på den valgte konkurrences kampe/medlemmer.
Hardcodede tekster ("VM Tips Kuponen 2026", VM-specifikke `Info`/`Prizes`)
erstattes af konkurrencens navn; `Prizes`-indhold gøres til et fritekstfelt på
konkurrencen (admin kan redigere) eller udelades i v1.

### Nye sider

| Side | Indhold |
|------|---------|
| `/opret` | Navn + turneringsvalg (dropdown over de 12 gratis football-data-koder med sæsondatoer + "Manuel/andet") → `create_competition` → viser invitationslink |
| `/join/{token}` | Viser konkurrencenavn via `get_competition_by_token`. Er man logget ind: join-knap. Ellers: login/signup-formular (genbrug fra `Home`) efterfulgt af auto-join |
| `/admin` | Kun synlig for admin af valgt konkurrence: kopiér/regenerér invitationslink, medlemsliste (+ fjern medlem), toggle påmindelser, "Afslut konkurrence"-knap, "Send afslutningsmail"-knap (se etape 6), og for `Source='manual'`: kamp- og resultatindtastning |

### GitHub Pages-detalje

`/join/{token}` er et deep link — GitHub Pages skal bruge SPA-fallback
(`404.html`, der redirecter til `index.html`). Tjek om `wwwroot` allerede har
det (NotFound.razor findes, men fallback-filen skal verificeres); ellers
tilføjes standard-tricket i deploy-workflowet.

## Etape 6 — Mails styret af admin

Admin kan ikke trigge GitHub Actions direkte fra appen (det ville kræve en
GitHub-token i klienten). Løsning: **mail-kø i databasen**.

```sql
create table "EmailRequests" (
  "Id" bigint generated always as identity primary key,
  "CompetitionId" int not null references "Competitions"("Id") on delete cascade,
  "Kind" text not null,             -- 'final_standings' (flere kinds senere)
  "RequestedBy" int not null references "Users"("Id"),
  "RequestedAt" timestamptz not null default now(),
  "ProcessedAt" timestamptz         -- null = venter; sat = udført (dubletværn)
);
```

- RPC `request_email(competition_id, kind)` — kun admin; afviser hvis der
  allerede ligger en uprocesseret/processeret request af samme kind (dublet-
  beskyttelsen, som den nuværende afslutningsmail mangler).
- **`notify-missing-predictions`** (scriptet): ombygges til at loope over
  aktive konkurrencer med `RemindersEnabled`, og kun maile medlemmer af den
  konkurrence, kampen hører til. Mailtekst bruger konkurrencens navn.
  Schedule-blokken genaktiveres (den no-op'er billigt uden aktive
  konkurrencer).
- **`send-final-standings`** (scriptet): ombygges til at læse `EmailRequests`
  (kind='final_standings', ProcessedAt is null), sende pr. konkurrence
  (stillingen beregnes pr. konkurrence — pagineringen fra juli-fixet
  genbruges) og stemple `ProcessedAt`. Workflow'et får et schedule
  (fx hvert 15. minut) i stedet for kun manuel dispatch.
- Afsender forbliver den centrale iCloud-konto (`SMTP_*`-secrets). OBS:
  ~500 mails/dag-loft og afsenderen er Kaspers adresse uanset hvem der er
  admin. Acceptabelt i venne-skala; skift til Resend/Postmark er en isoleret
  senere forbedring (kun scripts + secrets berøres).

## Etape 7 — Udrulningsrækkefølge (vigtigt: gammel klient må ikke knække undervejs)

1. **DB-migration A (additiv):** nye tabeller, RPC'er, `Matches.CompetitionId`
   som *nullable*, backfill af VM-data, derefter `not null`. Ingen
   RLS-ændringer endnu → den kørende app påvirkes ikke.
2. **Edge-funktion v2** deployes + pg_cron genschedules permanent
   (`select cron.schedule('sync-matches', ...)` som i REAKTIVERING.md §3).
   Uden aktive konkurrencer no-op'er den bare.
3. **Blazor-ombygningen** (etape 5) merges og deployes samlet.
4. **Scripts + workflows** (etape 6) opdateres; reminder-schedule genaktiveres.
5. **DB-migration B (RLS-stramning, etape 3)** køres til sidst, når ingen
   gamle klienter er i omløb (Blazor WASM cacher aggressivt — bump
   service-worker/cache-version i deploy).
6. `docs/REAKTIVERING.md` erstattes af en kort "drift"-note (guiden bliver
   overflødig — oprettelse sker i UI'et).

## Test inden lancering

- Supabase **branch database** til migrations-generalprøve (migration A + B
  mod en kopi), eller som minimum: kør migration A i prod (additiv = ufarlig)
  og RLS-migrationen på en branch først.
- End-to-end-manuel test: opret testkonkurrence med `Source='manual'` (ingen
  API-forbrug), join via link fra inkognitovindue med ny testbruger, tast
  resultat, se point og stilling, kør reminder-script med `ONLY_EMAIL`.
- `get_advisors` (security) køres efter RLS-migrationen.

## Åbne beslutninger

1. **Hvem må oprette konkurrencer?** Planen antager: alle med login (det er
   din Supabase/API-kvote, der bruges). Alternativ: en `allow_create`-flag på
   Users, som kun du kan sætte.
2. **Superliga-behovet:** dækket via `Source='manual'` (admin taster
   resultater). Godt nok, eller skal der findes en betalt/alternativ API?
3. **Prizes/Info-siderne:** fritekst pr. konkurrence (admin-redigerbar) eller
   udelades i v1?
